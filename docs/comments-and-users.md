# 评论与用户

## 评论模型

评论存储在 `gb_comments`。`type` 可为普通评论、Pingback 或 Trackback；`status` 使用：

| 状态 | 说明 |
|---|---|
| `approved` | 已通过，可公开展示 |
| `waiting` | 待审核 |
| `spam` | 垃圾评论 |

`parent` 指向父评论，`ownerId` 记录内容作者，`authorId` 在登录用户评论时记录用户 ID。评论条数缓存到内容的 `commentsNum`，审核状态变更和删除后会同步更新。

## 前台提交流程

普通评论大致经过以下步骤：

1. 校验评论 CSRF 令牌、来源页和内容评论开关。
2. 检查内容是否超过自动关闭天数。
3. 获取可信客户端 IP，并执行 IP 黑名单和提交间隔检查。
4. 登录用户直接使用账户昵称、邮箱和网站；访客读取表单字段。
5. 校验必填项、保留昵称、停用词和父评论。
6. 根据审核模式决定 `approved`、`waiting` 或 `spam`。
7. 调用评论保存钩子并写入数据库。
8. 更新评论数，为访客保存身份 Cookie；待审核评论额外保存自可见签名 Cookie。

已登录管理员/编辑或内容作者属于可信评论者，不需要重新填写昵称、邮箱和网站，也不会被普通访客的审核和提交间隔规则阻塞。

## 审核模式

后台“评论处理”组合提供三种主要模式：

| 模式 | 行为 |
|---|---|
| 不审核 | 正常评论直接通过，命中反垃圾规则的除外 |
| 全部审核 | 非可信评论者提交后进入 `waiting` |
| 已审核作者 | 作者昵称和邮箱曾有通过记录，或属于可信登录用户时通过；其他进入 `waiting` |

配套选项包括：

- 评论是否要求邮箱或网站。
- 评论来源页检查。
- 反垃圾总开关、停用词和 IP 黑名单。
- 同一来源提交间隔及其开关，默认 60 秒。
- 自动关闭旧文章评论的天数，`0` 表示不自动关闭。
- 评论最大嵌套层级，处理器限制为 2 到 7。
- 前台/后台每页评论数、排序和分页起始页。

审核模式会先计算默认状态，`comment.before_save` 钩子仍可修改最终 `SaveCommentInput.Status` 或返回错误阻止保存。

## 未审核评论自可见

访客提交待审核评论后，系统把评论 ID 写入带 HMAC 保护的 `unapproved_comment` Cookie：

- Cookie 最多保留最近 20 个 ID。
- 有效期为 30 天。
- 查询评论时，这些 ID 会加入当前访客可见范围。
- 其他访客看不到该待审核评论。
- 登录的内容所有者也可以查看所属内容的待审核评论。

这个机制只提供可见性，不代表评论已审核，也不能用于管理操作。清除 Cookie 或更换浏览器后，未登录访客无法继续证明自己是提交者。

## 评论正文渲染

评论可选择普通文本、允许部分 HTML 或 Markdown：

- Markdown 模式走评论 Markdown 钩子和默认 Markdown 渲染。
- 配置允许 HTML 标签时，核心按白名单清理并可为链接添加 `nofollow`。
- 默认普通文本会转义用户输入。
- `comment.before_render` 可修改原文，`comment.after_render` 可过滤最终 HTML。

插件返回的 `template.HTML` 是受信任内容。接管评论解析时必须自行清理访客输入。

## 评论管理

后台评论列表默认可查看全部状态，并支持单项和批量通过、待审核、标记垃圾和删除。后台回复与编辑使用独立内嵌表单：

- 回复只创建一条新的子评论。
- 编辑可修改昵称、邮箱、网站、IP 和正文。
- 状态变更会校验最终值只能属于 `approved`、`waiting`、`spam`。
- 删除父评论时服务层会处理子评论重新挂接，避免悬空层级。

单项、批量和清空垃圾评论都通过相同的钩子辅助函数，不会绕过评论生命周期钩子。

## 评论头像

头像默认根据规范化邮箱的 MD5 哈希生成。`avatar_url_template` 为空时使用内置地址；自定义模板支持：

```text
https://weavatar.com/avatar/{hash}?s={size}
```

可用占位符为 `{hash}` 和 `{size}`。评论头像还会应用头像开关和等级参数。插件可使用 `comment.avatar` 修改或清空最终 URL。

不要把原始邮箱拼接进公开头像 URL；自定义模板应只使用哈希占位符。

## 用户角色

角色按权限从低到高排列：

| 角色 | 典型能力 |
|---|---|
| `visitor` | 登录和个人配置，不具备内容管理权限 |
| `subscriber` | 登录和个人配置 |
| `contributor` | 创建和编辑自己的文章、使用附件 |
| `editor` | 管理文章、页面、分类标签和评论 |
| `administrator` | 管理用户、站点选项、插件、主题、WAF及全部后台功能 |

具体处理器仍会组合资源所有权检查。例如贡献者不能编辑其他作者内容，页面编辑要求 `editor`。新增后台路由时必须显式调用对应鉴权方法，不能只依靠侧边栏是否显示。

## 登录与会话

密码使用 bcrypt 保存。登录成功后签发 HMAC-SHA256 会话 Cookie：

- 默认有效期 7 天。
- Cookie 名由前缀和 `gopherink_session` 组成。
- `Secure` 和 `SameSite` 由站点配置控制。
- 会话载荷包含用户 ID、到期时间和用户 `authCode` 版本。
- 修改密码等需要撤销会话的操作会更新 `authCode`，使旧 Cookie 失效。

会话 Cookie 是签名数据，不是服务端 Session ID。`auth_secret` 必须保密且在实例间一致；更改它会让所有现有会话和相关签名失效。

后台无法登录时，可使用 `gopherink user list` 查询用户 ID，再通过 `gopherink user reset-password --id ID` 重置密码。命令直接读取启动配置指向的数据库，使用与后台相同的 bcrypt 和 `authCode` 轮换逻辑，但不会启动 HTTP 服务。完整用法见 [安装与配置](installation-and-configuration.md#用户应急命令)。

## CSRF

核心按用途生成每日 HMAC 令牌，包括登录、注册、安装、评论、后台和部分公开动作。令牌用途不应混用；新增 POST/PUT/DELETE 管理入口时应沿用后台 CSRF 校验，而不是只检查登录 Cookie。

## 评论钩子

普通评论、后台回复/编辑、Pingback 和 Trackback 都会经过通用 `comment.before_save` / `comment.after_save`。操作值分别为：

```text
comment, reply, edit, pingback, trackback
```

回复和编辑还会触发各自的前后钩子；状态变更、删除、渲染和头像也有独立钩子。完整 payload 和示例见 [插件与钩子开发](plugins-and-hooks.md)。
