# 架构与数据模型

## 设计概览

GopherInk 是一个服务端渲染的 Go CMS。程序将后台模板、后台静态资源、内置主题和内置插件编译进单个二进制；运行时通过数据库保存内容、用户和选项，通过本地目录保存默认附件与日志。

主要分层如下：

1. `cmd/gopherink` 读取启动配置、打开数据库、初始化 Schema，并组装服务与 HTTP 应用。
2. `core/handlers` 负责路由、鉴权、CSRF、表单解析、主题渲染、PJAX 响应和 WAF。
3. `core/services` 实现内容、评论、分类标签、用户和选项的业务规则。
4. `core/models` 定义持久化结构、数据库方言及当前版本 Schema。
5. `core/plugin` 提供构建时插件/主题注册、配置 Schema、路由、后台菜单和钩子调度。
6. `pkg` 提供认证 Cookie、安全 HTTP 客户端、国际化、图片处理、Markdown 渲染和 Slug 工具。

处理器不会把 SQL 细节扩散到模板中；主题通过处理器整理后的模板数据读取内容。插件可以在业务生命周期的钩子中修改载荷或接管特定默认实现，但不能替代数据库 Schema 初始化流程。

## 启动流程

`cmd/gopherink/main.go`、`runtime_config.go` 的启动顺序为：

1. 合并内置默认值、`data/config.json`、环境变量和 CLI 临时参数；首次交互式启动可先写入 JSON。
2. 校验数据库参数、上传目录、绑定 IP/端口、客户端 CIDR 和 TLS 路径；启用 TLS 时在数据库初始化前加载证书密钥对。
3. 打开写数据库；配置读库时再打开独立读连接。
4. 调用 `models.InitializeSchema` 创建当前版本表结构。
5. 初始化默认选项和认证密钥。
6. 在空数据库中创建显式配置的初始管理员，或启用 Web 安装向导。
7. 确保默认分类存在。
8. 以显式 `DataDir` / `UploadDir` 构造 HTTP 应用，在 WAF 外层安装直接来源 CIDR 访问控制，并按启动配置开始 HTTP 或 HTTPS 监听。

`config` 和 `user` 子命令会在进入完整启动流程前分流：前者只写 JSON，后者只连接数据库执行用户查询或密码重置。它们不会初始化主题、插件、路由和后台选项。

当前数据库版本为 `1`，新安装直接创建最终结构。项目不保留早期开发阶段的迁移链，因此升级源码前应自行备份数据库，并确认新版本是否改变 Schema。

## 构建时扩展

插件和主题由 `cmd/gopherink/plugins.go` 空白导入，包的 `init` 函数调用 `plugin.Register` 或 `plugin.RegisterTheme` 完成登记：

```go
import (
    _ "github.com/Chocola-X/GopherInk/plugins/sitemap"
    _ "github.com/Chocola-X/GopherInk/themes/default"
)
```

这意味着：

- 新增、删除或修改插件/主题后必须重新编译并重启。
- 后台启停插件只控制当前二进制中已登记代码的钩子和路由，不会装载新的机器代码。
- 主题切换只在已编译主题之间进行。
- 不支持运行时安装、Go `.so` 热加载或卸载。

## 数据库与读写分离

支持 SQLite、MySQL/MariaDB 和 PostgreSQL。`core/services/dbrouter.go` 将数据库能力抽象为统一接口：

- `Exec` 和 `Begin` 始终使用写库。
- `Query` 和 `QueryRow` 默认使用读库；未配置读库时回退到写库。
- 需要“写后立即读”一致性的流程使用 `services.WithWriter(ctx)` 强制查询写库。
- SQL 占位符和少量方言差异由 `core/models/query.go` 处理。

读写分离依赖外部数据库复制。GopherInk 不负责建立复制链路，也不保证从库零延迟；编辑后立即读取、身份认证等强一致场景必须走写库。

## 核心数据表

当前 Schema 包含八张核心表：

| 表 | 责任 |
|---|---|
| `gb_contents` | 文章、独立页面、附件以及编辑草稿 |
| `gb_users` | 用户、角色、密码哈希和会话版本码 |
| `gb_options` | 站点选项与按用户保存的个人选项 |
| `gb_metas` | 分类和标签 |
| `gb_relationships` | 内容与分类/标签的多对多关系 |
| `gb_comments` | 评论、Pingback 和 Trackback |
| `gb_fields` | 内容自定义字段 |
| `gb_revisions` | 文章和页面的历史修订快照 |

### 内容记录

`gb_contents.type` 区分 `post`、`page` 和 `attachment`。文章和页面共享保存、草稿、修订、自定义字段及附件父级的大部分逻辑。

关键字段：

- `cid`：数据库主键，也是内容内部标识。
- `slugId`：面向公开固定链接的递增数字标识。
- `slug`：可选的自定义固定链接片段；公开 URL 优先使用它。
- `draftOf`：编辑草稿所属的已发布内容 ID；独立未发布草稿为 `0`。
- `status`：发布、草稿、待审核、私密或隐藏等状态。
- `parent`：附件所属内容 ID；普通文章和页面通常为 `0`。

历史修订不再伪装成 `gb_contents` 中的内容，而是保存在 `gb_revisions`。因此普通内容查询不会把修订数量计入文章数。

### 选项记录

`gb_options` 使用名称和用户维度保存配置。站点级选项使用公共用户维度；插件站点配置使用 `plugin:<name>`，个人配置使用 `plugin:<name>:personal`，主题配置使用 `theme:<name>`，值通常为 JSON。

不要让主题或插件直接假设某一数据库驱动的 JSON 类型。当前配置通过普通文本读写并在应用层编解码。

## HTTP 请求链

公开请求的大致顺序为：

1. WAF 解析客户端 IP、检查封禁和分类限流。
2. 判断插件路由、公开 URL 索引和缓存是否可直接处理。
3. 路由到前台、后台、附件、订阅或兼容接口处理器。
4. 处理器执行认证、权限、CSRF 和输入校验。
5. 服务层读写数据库。
6. 已启用插件的相关钩子按优先级运行。
7. 前台由当前主题渲染，后台由嵌入模板渲染。

后台管理员已登录时，后台路径会绕过 WAF 频率与封禁规则，但不会绕过权限和 CSRF。公开插件路由会被 WAF 识别，避免被 URL 索引误判为不存在；这不等于插件路由自动具备后台鉴权。

## 渲染与可信边界

正文渲染由 `pkg/render` 提供：

- 默认 Markdown 引擎为 Goldmark，启用 GFM、自动标题 ID 和硬换行。
- `<!--markdown-->` 和 `<!--plaintext-->` 可覆盖内容的默认格式。
- HTML 模式按受信任内容输出；普通文本模式会转义后生成段落和换行。
- 插件解析钩子返回的 `template.HTML` 被视为受信任输出。

插件接管 Markdown、评论渲染或注入前端 HTML 时，必须自行转义或净化访客可控内容。钩子系统不会二次清理已声明为 `template.HTML` 的结果。

## 缓存和进程状态

公开页面缓存、WAF 计数器、临时封禁和登录封禁保存在进程内存中：

- 重启程序会清空这些状态。
- 多实例部署时，各实例默认不共享计数和缓存。
- 内容写操作会使当前实例的公开 URL 索引和页面缓存失效。
- WAF 事件本身写入 `data/waf.log`，不写入数据库。

需要跨实例一致限流或共享缓存时，应在反向代理层或后续扩展中接入共享存储；当前核心没有 Redis 适配层。

## 修改核心时的检查清单

- 增加公开路由时，检查 WAF URL 索引和插件路由分支。
- 增加写操作时，检查 CSRF、权限、缓存失效和写库读取。
- 改动内容保存时，同时检查新建草稿、已发布编辑草稿、自动保存、发布合并和修订。
- 改动附件时，同时检查本地存储钩子、远程存储钩子、缩略图和内容删除策略。
- 返回 `template.HTML` 或脚本标签时，明确可信来源并避免拼接未转义输入。
